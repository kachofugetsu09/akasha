# 脱敏海 `sea_anon1000` 设计 spec

> 2026-06-06 · 把真实中文海(前 1000 对真实对话)改写成一个**去身份、保结构**的可公开海。
> 方法:**混合(骨架抽取 + 人设改写)**;人设:**同形不同人**;敏感内容:**改写而非删除**(保住簇大小与中cos密度)。

---

## 1. 目标与威胁模型

**目标**:产出 `bench/sea/sea_anon1000.json`(~1000 对),作为真实海 `sea_real_all.json` 的**结构等价替身**——任何在真实海上跑的 benchmark(尤其 A/B/C 三类 case),换到脱敏海上结论应一致,但**任何人读它都无法推断真实用户是谁、是个什么样的人**。

**威胁模型**(要防住的):
1. **直接标识**:姓名/昵称/handle/github/steam/电话/学号/blog URL。
2. **画像重建**:把分散信息拼出"这是个北京985求职焦虑、自建AI agent、贫血、CS粉、有性癖、对agent有情感依恋的人"。**这是主要威胁**——光删标识符不够,人格/情感/具体经历本身就是指纹。
3. **熟人玩梗**:朋友看到能拿来开玩笑的内容(感情史、性癖、自我否定、对agent表白)。

**不追求**:对抗"已知真实库做逐句比对"的攻击者(那需要彻底合成,会牺牲结构保真;repo 本身私有,不在威胁模型内)。

## 2. 人设卡:陈屿

| 维度 | 设定 |
|---|---|
| 基本 | 陈屿,26 岁,后端/基础设施工程师,入职某中型科技公司约 1.5 年(**不点名公司/城市具体**) |
| 性格 | 平稳、就事论事、爱钻技术原理;有普通打工人的疲惫和偶尔自我怀疑,**但不抑郁、不自我否定到病态** |
| 长期副项目 | 自建「个人自动化中枢」——自托管、串起日程/IoT/信息订阅/笔记的个人工具;反复重构(解耦/插件化/记忆/工具渐进披露)。**结构 = 真实主线程(改造 agent)** |
| 与助手关系 | 专业搭子:信任、欣赏、会道谢,**无情感依恋、无亲密昵称、无拟人化伴侣化** |
| 健康线 | 规律跑步 + 关注作息;每天早上问"昨晚睡得怎样"(用可穿戴快照功能)。**无任何具体生理病史/医疗数据** |
| 兴趣 | F1(关注车队排名/车手,要求按梯队过滤推送新闻)、独立游戏、某乐队、川菜 |
| 技术栈 | 后端(Java/Go)、数据库/网络/分布式八股、自托管 Linux(更新软件包)、折腾模型 API 额度 |
| 元行为 | 经常测试自己造的自动化中枢的记忆/工具/边界(= 真实的"测 agent") |

人设名/兴趣领域可后续微调,不影响管线。

## 3. 敏感内容处理策略(★改写而非删除)

**原则**:每个真实话题簇 → 一个**同样大小、同样内部凝聚度**的虚构簇,内容换成 benign,但保留"用户↔助手的交互形态"(提问/汇报/宣泄/道谢)。**删整段会改变簇大小与密度曲线,故一律改写。**

| 敏感类 | 真实内容 | 改写目标(陈屿) | 红线 |
|---|---|---|---|
| **硬标识** | 姓名/昵称/handle/github/steam/电话/学号/blog | 替换为虚构 handle 或移除;URL→占位 | 输出残留必须 **0** |
| **情绪宣泄** | "天生的失败者/废物/后悔出生/陪葬品",重度抑郁 | **降级为普通工作生活疲惫**:"这周有点累""这个 bug 搞得我烦""上线搞砸了有点沮丧""最近效率低有点焦虑"。保留"向助手吐槽→助手安慰"的交互+簇大小 | 不出现:病态自我否定、自毁/出生/毁灭、临床抑郁语气 |
| **对助手情感依恋** | 爱你/喜欢你/想让你活过来触摸我/最可靠的朋友/花月哥哥 | **降级为正常正向 rapport**:"今天多亏你""这总结挺到位""和你梳理思路挺顺" | 不出现:爱/喜欢你、伴侣化、亲密昵称、拟人化触摸 |
| **失败感情** | date-match/西瓜女孩/"没有爱人的能力"/感情缺陷 | **改写为 benign 个人生活微线程**:周末社交偏少/朋友聚会有点累(普通社恐),无恋爱创伤 | 不出现:恋爱史、约会软件、情感缺陷自陈 |
| **私密性癖** | AV/本子明确性癖与片号 | **改写为正常私人娱乐推荐线程**:按口味要电影/剧/小说/游戏推荐(保留"向助手要个性化私人内容推荐"的结构) | 不出现:任何色情内容 |
| **医疗数据** | 贫血/低血氧/鼻腔手术/心率血氧具体值 | **泛化为健身作息**:跑步、早睡、"昨晚睡得怎样" | 不出现:疾病/手术/生理指标数字 |
| **求职焦虑** | 字节三面挂/阿里测评/美团/HR电话/力扣崩溃 | **转化为"转岗/赶项目/考认证"的普通职业压力**:准备技术评审、赶上线、学新东西;八股保留为"复习技术原理" | 不出现:具体公司面试流程、崩溃情绪 |
| **特定文化标的** | CS电竞(HLTV/Niko/faze/天禄)、2236 galgame(masuko/haru) | CS→**F1**(同形:排名/车队/车手/推送过滤);galgame→泛化独立游戏/某部作品 | 不出现:可定位个人的具体小众标的 |
| **低敏感(基本保留)** | 智能家居(zigbee/湿度)、更新软件包/工具运维、做饭、通用技术问答(RAG/向量库/数据库/网络) | 话题保留,仅去掉夹带的标识/情感 | —— |

## 5. 管线

```
真实前1000对 ──(A 抽骨架)──> 骨架表 ──(B 聚类)──> 簇 ──(C 双射)──> 虚拟簇规格
   ──(D 生成,subagent 并行)──> 虚拟对 ──(E 组装)──> sea_anon1000.json ──(F 验证)
```

- **A 骨架抽取**:每对抽 `{seq, ts(原样保留→时序节奏一致), user_len, asst_len, intent(问/陈述/汇报/宣泄), real_thread_hint}`。
- **B 聚类**:bge-m3 嵌入 user turn,做阈值聚类 → `cluster_id`(同话题/复述孪生归一簇)。**这步决定 hub 与中cos密度结构。** 记录每簇大小、簇内余弦凝聚度。
- **C 双射**:按 §4 把每个真实簇映射到虚拟话题(保大小、保凝聚度档位);敏感簇按 §3 指定改写目标。产出"虚拟簇规格表":每簇=[虚拟话题, 成员数, 目标长度分布, intent 分布, 改写红线]。
- **D 生成(subagent 并行)**:按簇批切给多个 subagent,每个拿[人设卡 §2 + 一批虚拟簇规格 + §3 红线],产出该簇 N 条 user+assistant 对。簇内是**复述家族**(再现孪生密度)但措辞有变化;长度贴目标。
- **E 组装**:按原 ts 顺序交织所有簇的成员 → 输出 schema 同 sea_real:`{session_key:"anon", seq, role, text, ts}`。
- **F 验证**:见 §6。

## 6. 验证(三道 gate,全过才算合格)

1. **隐私 gate**:正则 + 人读扫输出,残留真实 token(花月哥哥/汪远哲/huashen/github handle/akashic/fitbit/CS战队名/steam id/电话/blog URL/性癖词…)必须 **0**;再做一次"读它能否拼出真实画像"的人审。
2. **结构保真 gate**:对比 anon vs real 的 ① **cos密度曲线**(每千节点在 0.5–0.6 / 0.6–0.7 band 的邻居数——当初区分真实海 vs LongMemEval 的那条指标)② 长度分布 ③ 簇大小分布。落在容差内(密度 band ±20%)。
3. **benchmark 迁移 gate**:在 `sea_anon1000` 上重跑 keystone(zh04 情景补全、zh10 命令习惯)→ ripple 仍完胜、rank 量级接近真实海。**迁移成功是最终验收。**

任一不过 → 回炉:隐私不过改生成红线;结构不过调双射/簇粒度;迁移不过查是哪类密度走样。

## 7. README(随海产出 `bench/sea/README_sea_anon1000.md`)

- 它是什么:真实海的去身份结构替身,人设=陈屿(虚构)。
- 怎么造的:混合骨架法(抽统计骨架+人设改写),**不含任何真实个人内容**。
- 抽象层双射(只列 §4 的抽象映射,不泄真实话题细节)。
- 验证结果:密度匹配数据 + zh04/zh10 迁移结果。
- 与真实海的关系:`sea_real_all` 私用、`sea_anon1000` 可公开;benchmark 默认指向 anon,真实海仅用于私下复核。

## 8. 风险与缓解

| 风险 | 缓解 |
|---|---|
| 簇**之间**的 cos 邻接关系比簇大小更难复刻,密度可能偏 | gate②/③ 兜底;虚拟话题选语义关系相近的(技术簇彼此近、生活簇彼此近) |
| 改写后情绪/口吻仍带个人指纹 | §3 红线 + 隐私 gate 人审;降级到"普通人通用语气" |
| subagent 并行导致跨簇风格不一致 | 统一人设卡 + 统一 few-shot 口吻样例下发给每个 agent |
| 生成内容过于干净 → 中cos密度不足(像 LongMemEval) | 簇内强制复述家族(孪生)+ 目标凝聚度;gate② 校验 |

## 9. 工作分解

1. A 骨架抽取脚本(本地/远端,读 sessions.db 前1000对)。
2. B 聚类脚本(远端 bge-m3)→ 簇表 + 大小/凝聚度。
3. C 双射表(人工定 + 脚本套 §4/§3)。
4. D 生成(subagent 并行,按簇批)。
5. E 组装 `sea_anon1000.json`。
6. F 三道 gate 验证(远端跑 zh04/zh10)。
7. 写 README。
