## 1. 方法

- 语料:`sessions.db` telegram 前 1000 对(2026-03-11 → 04-13,32 天)。

- 每簇量三个维度:
  - **规模 n**(hub 强度)
  - **时间分布**:跨天 span / 活跃天数 / 跨周数(全局视角的命根)
  - **内聚度**:簇内平均成对余弦(高=紧密复述 echo;低=同话题但内容多样)

## 3. 三个结构轴 → 对检索的含义

**轴一·规模(hub 强度)**:n 越大,probe 命中后种子越多,但也越可能污染种子池(真实库失眠 88 孪生那种)。top5(19/68/17/28/2)是绝对主hub。

**轴二·时间分布(全局视角的前提)**:**15 个簇满足 n≥15 且跨≥3 周**(19,68,17,28,2,5,26,66,54,16,35,6,20,9,31)。这些是"同一话题在整段时间里反复出现"的线 —— 全局视角 case 只能建在这种线上,因为要量的就是"召回能否横跨多个时间段"。

**轴三·内聚度(echo vs 内容,决定 dense/ripple 谁占便宜)**:
- 关键观察:**主hub 内聚度多在 0.59–0.67,普遍 < dense 种子阈值 0.675**。意味着这些簇**不是铁板复述**,而是"同话题、措辞/内容有变化"。所以 probe 给定时,dense 只能勾到 cos 最高的一小撮(往往挤在某个相似/时间片),**勾不全整条时间线**。
- **紧密簇**(68睡眠0.656、54家居0.705、19情绪0.673):成员像"反复问同一句/同一类",dense 容易捞回一串近义**echo(内容空)**;ripple 沿边能带出每段背后的**实际内容**。→ 反差点 = "dense 给重复的问句,ripple 给跨时间的实际历史"。
- **松散簇**(17架构0.592、28求职0.587、5重构0.609):成员同话题但每次内容不同,本身就是一条跨时间演进线;dense 按 cos 只能咬住最像的几条,ripple 靠质量/跨激活摊到各周。→ 反差点 = "dense 给某一时段切片,ripple 给跨周 pattern"。

## 4. 对 B类「全局视角」benchmark 的结论

1. **指标必须换成时间维**(不是 A类的低cos facet 召回):
   - `时间桶覆盖率`:把目标簇成员按周分桶,数 dense@k / ripple@k 各命中几桶。
   - `时间跨度`:召回相关项的 max(ts)−min(ts)。
   - 赢的判据:ripple 覆盖更多桶 / 跨度更宽。
2. **内聚度决定 case 的"赢法",要两类都覆盖**:
   - 紧密 echo 型(睡眠/家居/情绪):证 "dense=重复问句、ripple=跨时间实际内容"。
   - 松散内容型(架构/求职/重构):证 "dense=某时段切片、ripple=跨周演进 pattern"。
3. **必须先实测验证**:dense 的时间桶分布到底窄不窄(近义复述可能本身就散在各周)。若 dense 也覆盖广,则区分点要落到"echo vs 内容",而非单纯桶覆盖。
